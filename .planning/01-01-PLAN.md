# 01-01-PLAN — Define shared V2 contracts and state transition rules

Last updated: 2026-04-03
Autonomous: true
User setup: none

## Objective
Create the shared V2 classroom contracts and explicit transition logic that establish the orchestrator as the source of truth for lesson flow.

## Scope
This plan should define the shape of V2, not complete the classroom rebuild.

## Files to Create or Modify
### Create
- `src/lib/classroom-v2/types.ts` or equivalent V2 contract file
- `src/lib/classroom-v2/transitions.ts` or equivalent pure transition/rule file
- `src/lib/classroom-v2/index.ts` or equivalent export surface

### Modify
- `src/components/VoiceSession.tsx` only if needed to reference or isolate the new V2 seam
- optionally add a small testable demo/helper module if the repo already has a pattern for local logic verification

## Implementation Requirements
1. Define `ClassroomState` covering:
   - `idle`
   - `tutor_rendering`
   - `tutor_speaking`
   - `student_answering`
   - `student_submitted`
   - `tutor_reviewing`
   - `ended`
2. Define `TutorAction` types supporting at least:
   - `speak`
   - `ask`
   - `board.writeProblem`
   - `board.writeText`
   - `board.highlight`
   - `board.arrow`
   - `board.underline`
   - `board.stepBox`
   - `board.clearTeacherLayer`
3. Define structured student evidence types for voice/board/both.
4. Implement explicit transition helpers that reject invalid events.
5. Document or encode the rules for when mic/board input are allowed.
6. Keep the engine contract independent from transcript text parsing and vendor-specific realtime details.

## Must-Haves
### Truths
- There is one clear source of truth for classroom turn/state in the new V2 module.
- Invalid transitions are represented and handled explicitly rather than silently accepted.
- The contracts clearly separate teacher actions from student evidence.
- The resulting module is reusable by later board and voice plans.

### Artifacts
- A V2 contracts file exists with typed classroom states and tutor actions.
- A transition rules file exists with explicit guard logic.
- Exports are organized so Phase 1-02 can build an orchestrator skeleton on top of them.

## Acceptance Checks
- Another engineer can read the new module and understand the intended V2 lesson loop.
- No new transcript-driven whiteboard logic is introduced.
- The repo still builds after the change.
