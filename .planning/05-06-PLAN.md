# 05-06-PLAN — Calmer board-state settling / end-of-beat hold polish in the isolated V2 preview seam

## Goal
Make teacher-owned beats in the isolated V2 preview feel like they land, settle, and hold briefly before the next handoff so the board reads less like a sequence of state flips and more like an intentional teaching surface.

## Scope
- Stay narrow and browser-local inside the isolated V2 preview seam and canonical V2 whiteboard presentation layer.
- Reuse the existing preview pacing/reveal seam instead of introducing a new scheduler or backend timing model.
- Add tiny reviewable hold/settle cues only for teacher-owned beats after the main reveal has effectively landed.
- Preserve debug/inspection surfaces and keep student drawing/tools behavior unchanged.

## Planned implementation
1. Update `src/features/classroom-v2/ClassroomV2Preview.tsx` to derive a preview-only teacher hold snapshot from the existing pacing model:
   - whether the current beat is in a brief landed/hold window
   - a hold label/detail for review surfaces
   - a small board-state status string that makes the intentional hold reviewable
2. Update the preview UI copy/cards in `ClassroomV2Preview.tsx` so teacher-owned beats can explicitly read as either revealing or intentionally holding once the board has landed.
3. Update `src/components/classroom-whiteboard/CanonicalWhiteboard.tsx` to accept and forward narrow preview-only board settle props.
4. Update `src/components/classroom-whiteboard/LayeredBoard.tsx` so the teacher layer and board shell can enter a calmer landed posture during that hold window:
   - reduce active motion emphasis once the reveal is effectively complete
   - add a subtle settled board posture / glow state for the brief hold
   - keep teacher content fully legible and stable while preserving student-layer behavior
5. Write `.planning/05-06-RESULT.md`, update `.planning/STATE.md`, and run `npm run build`.

## Out of scope
- Legacy classroom cutover
- VoiceSession default/runtime changes
- DB / persistence / parent dashboard changes
- New orchestrator/runtime contracts
- Broad animation systems or large refactors
