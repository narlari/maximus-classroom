# 01-02-PLAN — Implement orchestrator skeleton and event dispatcher

Last updated: 2026-04-04
Autonomous: true
User setup: none

## Objective
Build the first usable V2 orchestrator skeleton on top of the new classroom contracts so later voice and board work can plug into one authoritative lesson engine.

## Scope
This plan should create the minimal V2 runtime shape, not wire the full classroom UI.

## Files to Create or Modify
### Create
- `src/lib/classroom-v2/orchestrator.ts` or equivalent engine module
- optionally `src/lib/classroom-v2/adapters.ts` or equivalent interface file if that makes boundaries clearer

### Modify
- `src/lib/classroom-v2/index.ts` to export the new surface
- `src/lib/classroom-v2/types.ts` only if small contract additions are needed for orchestration clarity
- `src/lib/classroom-v2/transitions.ts` only if small helper additions are needed for readable engine flow
- optionally add a small test/demo helper if the repo already has a lightweight verification pattern

## Implementation Requirements
1. Create a thin orchestrator that owns a `ClassroomSession` instance and advances it through the guarded transition helpers.
2. Support a bounded event dispatcher API for at least:
   - start tutor turn
   - finish tutor rendering
   - finish tutor speaking
   - submit student evidence
   - begin tutor review
   - finish tutor review
   - end lesson
3. Expose state-derived adapter cues without binding to realtime/vendor payloads, including:
   - current classroom state
   - active turn
   - allowed student input flags
   - allowed next event types
4. Keep voice, board, logging, and student input behind explicit adapter seams or callback interfaces.
5. Preserve the architectural rule that transcript text is not the source of truth.
6. Keep the orchestrator small, readable, and obviously reusable by later Phase 1/2 work.

## Must-Haves
### Truths
- There is now a concrete V2 engine surface above raw transition helpers.
- Another engineer can see where voice and board will plug in without mixing those concerns into the core engine.
- Invalid transitions still fail explicitly rather than being hidden in UI logic.
- The orchestrator does not deepen legacy transcript-first behavior.

### Artifacts
- A V2 orchestrator module exists.
- The module can accept events and return updated session/runtime state.
- Export surface is clean enough for Phase 1-03 to wire an isolated V2 integration seam.

## Acceptance Checks
- Another engineer can instantiate the orchestrator and walk a lesson loop without touching browser/audio code.
- Allowed student input and allowed next events are derived from classroom state, not duplicated in consumers.
- Repo still builds after the change.

## Notes for Rex
Favor boring architecture. This is the bridge between pure contracts and the future classroom UI. Avoid clever abstractions, framework churn, or touching legacy live-session code unless a tiny seam is absolutely necessary.
