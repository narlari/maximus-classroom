# 06-09-PLAN — Targeted beat confirmation cue inside preview

## Goal
Add one tiny preview-only cue that tells Sung exactly what targeted review beat is currently open, what to look for to confirm it, and what the immediate follow-up beat is.

## Why this slice
06-07 and 06-08 made it easy to jump into one exact review beat and return with a report, but once preview opens Sung still has to interpret the current preview state and mentally map it back to the requested beat. A tiny in-preview confirmation cue removes that extra interpretation.

## Scope
- Keep legacy `VoiceSession` as the default classroom runtime.
- Keep everything browser-local and review-only.
- Stay local to `ClassroomV2Preview` and existing preview/review helper logic.
- Add a small targeted-beat cue card that:
  - names the current beat in plain language
  - says what visible condition confirms that beat
  - says what follow-up beat comes next
- Reuse existing active beat / checklist state instead of adding new persistence.
- Update planning/state docs.

## Non-goals
- No backend/schema/database changes.
- No default cutover to V2.
- No persistence model changes.
- No broader preview redesign.
- No changes to runtime orchestration behavior.

## Validation
- Launch preview from the real classroom route into a specific review beat.
- Confirm preview shows a compact beat cue describing the open beat, the visible confirmation signal, and the next follow-up beat.
- Confirm the cue updates if the preview advances beyond the targeted beat.
- Confirm non-targeted preview launch still behaves normally.
- `npm run build` passes.
