# 06-08-PLAN — Targeted review return-path reporting

## Goal
Make the route-to-preview targeted beat review loop close more cleanly by returning from preview with a tiny route-level report that says what beat was inspected and whether that beat is now confirmed.

## Why this slice
06-07 made direct beat launch possible, but after returning to the real classroom route Sung still has to infer what was just reviewed by looking at the remembered checklist. A tiny return-path report removes that ambiguity without changing runtime defaults.

## Scope
- Keep legacy `VoiceSession` as the default classroom runtime.
- Keep everything browser-local and review-only.
- Reuse the existing real route card and query-param launch/return seam.
- Add a preview return link that carries a tiny review report back to the classroom route.
- Show a short route-level report card/banner describing the just-reviewed beat and whether it is now confirmed.
- Update planning/state docs.

## Non-goals
- No backend/schema/database changes.
- No default cutover to V2.
- No legacy `VoiceSession` runtime behavior changes.
- No new persistence model.
- No broad preview redesign.

## Validation
- Launch a targeted review beat from `/classroom/<studentId>`.
- Return to the default classroom from preview and confirm the route shows a concise report naming the reviewed beat.
- Confirm the report distinguishes between a confirmed beat and an in-progress / not-yet-confirmed beat.
- Confirm the normal remembered review card still works and legacy `VoiceSession` remains the default outside explicit preview mode.
- `npm run build` passes.
