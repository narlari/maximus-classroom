# 06-13-PLAN — Targeted beat projected review progress inside preview

## Goal
Add one more tiny preview-only aid to the targeted-beat cue so Sung can immediately see what the review run looks like after this checkpoint, not just what action to take right now.

## Why this slice
06-12 answered “what should I do next?” but Sung still has to infer whether confirming this beat meaningfully advances the run or finishes it. A compact projected-progress panel keeps the same targeted-review seam, stays browser-local, and makes the review path more actionable without widening scope.

## Scope
- Keep legacy `VoiceSession` as the default classroom runtime.
- Keep everything browser-local and review-only.
- Stay local to `ClassroomV2Preview` targeted-beat cue logic/UI.
- Extend the existing targeted cue with a compact `Review progress after this beat` panel derived only from:
  - requested targeted beat
  - current checklist completion state
  - whether the requested beat is active now
  - the next unresolved review beat when helpful
- Keep existing verdict, confidence signals, recommended action, confirmation, and follow-up content intact.
- Update planning/state docs.

## Non-goals
- No backend/schema/database changes.
- No persistence model changes.
- No legacy runtime behavior changes.
- No route-level redesign.

## Validation
- Launch targeted preview flows.
- Confirm the targeted cue now states the projected review progress after this beat.
- Confirm the projection reads correctly for open-now, already-confirmed, and advanced/not-yet-confirmed states.
- Confirm non-targeted launches are unchanged.
- `npm run build` passes.
